Walk me through your project architecture — why this folder structure?Med▼
The project follows feature-colocation. api/services/ isolates all API calls so pages never call axios directly. store/ holds only global auth state via Zustand — nothing else needs to be global since React Query handles server state. layouts/ separates shell chrome from page content. This means any developer can find any file in under 10 seconds.
Why did you choose React Query instead of Redux for server state?Hard▼
Redux is great for client state (UI toggles, auth). But server state has different needs: caching, background refetch, stale-while-revalidate, and deduplication. React Query gives all of this for free with staleTime, queryKey cache invalidation, and useMutation. Writing all of that in Redux requires thunks, reducers, loading flags, and error fields — far more boilerplate for the same result.
How does your authentication flow work end-to-end?Med▼
1. User submits login form → validated by Zod + React Hook Form. 2. Axios posts to /auth/login. 3. Token stored in localStorage and in Zustand store (isAuthenticated: true). 4. Every subsequent request: the axios request interceptor reads the token and adds Authorization: Bearer. 5. If any response returns 401: the response interceptor removes the token and redirects to /login. 6. React Router's ProtectedRoute checks Zustand — if no token, redirect to login before rendering.
What is your handleResponse() helper and why did you need it?Easy▼
The API was inconsistent — some endpoints return data directly as res.data, others wrap it as res.data.data. Instead of handling this in every service, I wrote one normalizer: res.data?.data ?? res.data. This is the "adapter" pattern — it isolates the inconsistency in one place so the rest of the app deals with clean typed data.
How did you handle form validation? Why Zod + React Hook Form?Easy▼
React Hook Form is performance-first — it uses uncontrolled inputs so the component doesn't re-render on every keystroke. Zod gives me schema-based validation with full TypeScript inference, so the same schema that validates the form also types the submitted data. I connect them with zodResolver. This means validation rules, types, and error messages all come from one schema — no duplication.
How did you implement lazy loading and why?Med▼
Each page component is wrapped in React.lazy() and dynamically imported. The router wraps each route in <Suspense fallback={<Loader />}>. This means the JS bundle for the Courses page is only downloaded when the user navigates there — the initial bundle is much smaller, so the app loads faster on first visit. Vite handles the code-splitting automatically based on the dynamic imports.
What would you improve if you had more time?Easy▼
Three things: 1. Pagination — currently fetching all records at once which won't scale. I'd add cursor-based or offset pagination with React Query's useInfiniteQuery. 2. Optimistic updates — right now delete/update waits for the server. I'd use React Query's onMutate to update the cache instantly, then roll back on error. 3. Better error boundaries — currently toast handles errors globally but I'd add React Error Boundaries per page for crash isolation.
Core React concepts — expect 2–4 of these in any interview.
What is the difference between useEffect and useLayoutEffect?Hard▼
useEffect runs asynchronously after the browser paints — it won't block the visual update. useLayoutEffect runs synchronously after DOM mutations but before the browser paints — useful when you need to measure DOM elements or prevent visual flicker (e.g. tooltip positioning). In 95% of cases use useEffect. Use useLayoutEffect only when you see a flicker.
Explain the React rendering cycle and when re-renders happen.Hard▼
A component re-renders when: 1. Its own state changes. 2. Its parent re-renders (even if props haven't changed, unless wrapped in React.memo). 3. A context value it consumes changes. 4. A hook it uses returns a new value. React then runs reconciliation — diffs the virtual DOM tree — and only commits actual DOM changes. The render phase is pure (no side effects); the commit phase applies changes and runs effects.
What is the difference between useMemo and useCallback?Med▼
useMemo memoizes a computed value — it re-runs the function only when dependencies change, returns the value. useCallback memoizes a function reference — it returns the same function instance between renders unless dependencies change. useCallback(fn, deps) is equivalent to useMemo(() => fn, deps). Use useCallback when passing callbacks to React.memo children so they don't re-render on every parent render.
What is prop drilling and how do you solve it?Easy▼
Prop drilling is passing data through multiple intermediate components that don't use it, just to reach a deeply nested child. Solutions: 1. React Context — good for global/semi-global state like theme, auth. 2. Zustand/state management — used in this project for auth. 3. Component composition — restructure so the component that needs the data is rendered closer to where the data lives. 4. React Query — for server data, any component can query directly without passing through parents.
Explain the key prop — why is it important in lists?Easy▼
The key prop tells React which items in a list have changed, been added, or removed. Without keys (or with index as key), React re-renders the whole list on change and can produce bugs — e.g. an input's focus or a component's local state can jump to the wrong item. Keys must be stable and unique. Using array index as key is wrong when the list can be reordered or filtered because the index changes even when the item doesn't.
What are controlled vs uncontrolled components?Med▼
Controlled: form input's value is driven by React state — every keystroke calls setState, React owns the value. Uncontrolled: the DOM owns the value, you read it via a ref when needed. React Hook Form uses uncontrolled inputs by default (via refs), which is why it's fast — no re-render on every keystroke. You'd use controlled when you need to validate or transform input in real time, or drive the input value programmatically.
Performance is a core skill for this role — know these deeply.
How did you optimize performance in this project?Med▼
Four main optimizations: 1. React Query staleTime — categories are cached for 5 min, so navigating back doesn't refetch. 2. Lazy loading — each page is a separate bundle chunk, so initial load is minimal. 3. React Hook Form — avoids controlled input re-renders. 4. Zustand over Context — Zustand only re-renders components that subscribe to the specific slice they use; Context re-renders all consumers on any change.
What is code splitting and how does Vite do it?Med▼
Code splitting breaks the JS bundle into smaller chunks loaded on demand. Vite uses Rollup under the hood — any dynamic import (import()) automatically becomes a separate chunk. React.lazy() wraps dynamic imports so React knows to load the chunk when the component is first needed. The result: a 50kb initial bundle instead of 500kb — critical for time-to-interactive on slower connections.
When would you use React.memo() and what are its pitfalls?Hard▼
React.memo() wraps a component to skip re-renders when props haven't changed (shallow comparison). Use it for: expensive render components that receive stable props, list items in a large list. Pitfalls: 1. If a parent passes an inline object or function as prop, it's a new reference every render and memo won't help — must use useMemo/useCallback. 2. The shallow comparison itself has a cost — don't memo cheap components. 3. It only compares props, not context or internal state.
What is the virtual DOM and how does reconciliation work?Hard▼
The virtual DOM is a lightweight JS representation of the real DOM. When state changes, React creates a new virtual DOM tree, diffs it against the previous one (reconciliation), and calculates the minimal set of actual DOM operations needed. React's diffing algorithm runs in O(n) by assuming: elements of different types produce different trees, and key props identify stable children across renders. Only the changed DOM nodes are updated — this is much faster than rebuilding the full DOM.
How would you handle a component that re-renders too often?Hard▼
Step 1: Diagnose with React DevTools Profiler — find which component renders and why. Step 2: If caused by a parent re-render with stable props → wrap child in React.memo. Step 3: If a callback prop is recreated every render → useCallback. Step 4: If a computed value is expensive → useMemo. Step 5: If context causes it → split context into smaller pieces or use a selector pattern (Zustand supports this natively). Step 6: Move expensive state down to the component that needs it — don't lift unnecessarily.

What is the difference between interface and type in TypeScript?Med▼
Both define shapes. Key differences: 1. interface supports declaration merging — two interfaces with the same name merge. type does not. 2. type can represent union types, intersection types, primitives, tuples — more powerful. 3. interface is better for defining object shapes (class contracts, API models) because it's extendable with extends. In this project: data models like Course use interface, utility types like form data use type.
What is a generic in TypeScript and where did you use them?Med▼
Generics let you write reusable code that works with different types while preserving type safety. In the project: the Table<T> component accepts any data type — columns render functions get correctly typed row data. The handleResponse<T> helper returns typed data. React Query's useQuery<Course[]> makes data typed as Course[] | undefined. Without generics you'd need a separate component for each data type.
What is the difference between any, unknown, and never?Hard▼
any: opt out of type checking — you can do anything with it, TypeScript won't complain. Avoid it. unknown: you don't know the type yet, but TypeScript forces you to narrow it before using it — type-safe alternative to any. Used for API responses, catch(error) values. never: a value that can never exist — used in exhaustive checks, functions that always throw, and conditional types that resolve to an impossible case.
How does Zod connect to TypeScript types?Easy▼
Zod schemas are both runtime validators and TypeScript type sources. z.infer<typeof schema> extracts the TypeScript type from the schema automatically — so if the schema changes, the type updates too. This means the validation rules and the TypeScript type are always in sync — no separate interface to maintain. In the project, type FormData = z.infer<typeof schema> is used for every form.

How did you approach UI/UX design decisions in this project?Easy▼
Three principles: 1. Loading states everywhere — skeleton loaders instead of layout shifts, disabled buttons during mutations to prevent double-submission. 2. Empty states — every list has a meaningful empty state with a call-to-action, not a blank screen. 3. Feedback — every mutation (create/update/delete) shows a toast so the user knows what happened. The Tailwind utility-first approach kept CSS consistent — spacing, color, and radius all follow the design token system.
What is Tailwind CSS and what are the tradeoffs vs traditional CSS?Easy▼
Tailwind is a utility-first CSS framework — you style with small single-purpose classes directly in JSX. Pros: no naming CSS classes, no dead CSS, co-located styles, design token consistency, tiny production bundle (PurgeCSS removes unused utilities). Cons: HTML gets verbose, no semantic class names for readability, long class strings can be hard to scan. For component libraries and dashboards it's excellent — for document-heavy sites traditional CSS may be cleaner.
How would you make this app fully accessible (a11y)?Hard▼
Current gaps and fixes: 1. Modals need role="dialog", aria-modal="true", focus trap (focus stays inside modal), and restore focus on close. 2. Tables need scope="col" on headers. 3. Icon-only buttons need aria-label. 4. Form errors need aria-describedby pointing to the error message. 5. Color alone shouldn't convey meaning — badges use both color and text. 6. All interactive elements must be keyboard navigable with visible focus rings.
How would you implement dark mode?Med▼
With Tailwind: enable darkMode: 'class' in config. Toggle a dark class on <html>. Use dark: variant on all color utilities. Store preference in localStorage, initialize from prefers-color-scheme media query on first load to respect system setting. For the component architecture: define CSS variables for all semantic colors in both :root and .dark — then components use variables, not hardcoded colors.
